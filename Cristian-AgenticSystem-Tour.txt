# CLAUDE.md — Agentic Design System Infrastructure

## What This Project Is

This is an **Agentic Design Systems Infrastructure** — a token-first, metadata-driven,
AI-native design system monorepo. It is NOT a traditional design system. It is several
interconnected systems running side by side, wired together so that an AI agent (Claude Code)

The core philosophy:
> "Agentic doesn't mean everything is an LLM. It means the system closes its own loop."

The agent reads indexes to navigate, acts only through skills (slash commands), and what
those skills generate — maps, audit reports, fix suggestions — loops back to the agent
that consumes them. That self-closing loop is what "agentic" means here.

## Architecture: 5 Layers + 1 Loop

The system is organized into 5 layers and 1 agentic feedback loop:

### 🟡 1. Tokenization Layer — "Where everything starts"
The foundation. Design decisions are encoded here as W3C Design Tokens.

- **`tokens/*.json`** — 348 tokens across 12 categories: colors, semantic colors,
  spacing, typography, text styles, radius, borders, motion, elevations, z-index,
  icon sizes, breakpoints. Two-layer color architecture: foundation palettes
  (moonstone, wisteria, mauve…) referenced by semantic roles. This is the START.
- **`Style Dictionary`** — The token compiler. Reads the W3C source JSON and emits
  platform outputs: CSS custom properties for web, JS constants with TypeScript
  declarations for programmatic use. Nx runs it automatically before any package
  that depends on tokens, so components can never build against stale values.
- **`variables.css`** — The compiled stylesheet every component references.
  Token names flatten to CSS custom properties: `{color.moonstone.500}` becomes
  `--color-moonstone-500`; semantic roles become `--foreground-base`,
  `--background-interactive-primary`. One change at the source = all 78 components follow.

### 🟢 2. Intent Layer — "What components declare"
Components declare their identity and intent here. Nothing is hardcoded.

- **`Component.module.css`** — Token-based CSS module. Every visual decision resolves
  to a `var(–token)`. CSS modules keep styles scoped per component while `var(–…)`
  references keep them tethered to the tokenization layer. This surface is what the
  token auditor patrols.
- **`Component.tsx`** — React component logic using CVA (class-variance-authority).
  Every prop axis (variant, size, color) maps to CSS module classes, with defaults.
  The TypeScript interface is the contract that metadata, docs, the prop glossary,
  and Figma generation all read from. One source → four consumers.
- **`Component.metadata.ts`** — Full AI-ready metadata per component: use cases,
  common composition patterns with code, anti-patterns with alternatives, accessibility
  notes, and AI hints. This is "API docs for agents" — it tells Claude when and how to
  use a component correctly, and what NOT to do. Also the source of truth for the
  human-facing Astro docs site.
- **`/scaffold-component`** (skill) — Components are NOT written from memory. One
  invocation produces the same five-file anatomy every time:
  `Component.tsx` + `Component.module.css` + `Component.metadata.ts` +
  `Component.test.tsx` + `index.ts`. The anatomy is a contract.
- **`Metadata Schema`** (v2.0) — The versioned schema every `.metadata.ts` must follow:
  required fields, valid categories and types, date formats, priority levels.
  Lives at `.claude/rules/components/metadata-schema.md`.
- **`Prop Glossary`** (`.ai/prop-map`) — The canonical prop vocabulary across
  71 web + 17 React Native components. Axes registry (size, variant, color, radius…
  with canonical values), an anti-synonym value glossary ("outline", never "outlined"),
  per-prop usage, and drift flags. Descriptive first — divergence is annotated for
  review, never silently migrated.

### 🔵 3. Indexing Layer — "How the agent navigates"
This layer makes the codebase navigable without codebase crawling.

- **`/codebase-index`** (skill, v1.2.0) — Automatically generates relationship and
  dependency maps for the entire component-based codebase. Auto-detects framework,
  scans every component, and emits TOON-format maps (30–60% more token-efficient
  than JSON): which component uses which, which tokens each consumes, and what
  external dependencies exist. This is what lets the agent navigate instead of explore.
- **`index_codebase.py`** — The deterministic engine inside `/codebase-index`.
  A pure scanner that walks every component file, extracts imports, usage, token
  references, and metadata adjacency, and emits the TOON maps — same input, same
  output, every run. The LLM decides WHEN to index; this script decides WHAT the
  index contains. The split is the heart of the architecture.
- **`.ai/index.toon`** — The design system index the agent loads once at session start:
  78 components, 46 with metadata, 348 tokens across 12 categories, 657 relationships
  mapped. Load once, keep in context, fetch component metadata on demand.

### 🟣 4. Orchestration Layer — "How the agent is shaped"
This is where Claude Code becomes a system-aware operator, not a general model.

- **`AI Agent`** (concept) — Claude Code is the runtime of the system. At session start
  it loads the project instructions and the codebase index, then operates through
  skills — scaffolding, auditing, indexing, syncing to Figma. The agent never
  improvises against the system: rules constrain it, the index informs it, and every
  skill reads and writes well-defined artifacts.
- **`Rules`** — The registry of project-specific rules in `.claude/rules/`. Each rule
  carries a `paths:` frontmatter, so it loads only when the agent touches matching
  files — the agent isn't pre-loaded with all of them, it is shaped by the relevant
  ones in context. Rules groups:
  - `tokens-system.md` — naming and semantic layers
  - `components/rules` — metadata schema and atomic hierarchy
  - `codebase-index/rules` — what the maps must compute
  - `skills-integration.md` — teaches the agent its own toolset
  - `best-practices rules` — general DS best practices

### 🩷 5. Figma Integration — "Code is the source; Figma is the render target"
Figma is NOT the source of truth. The code is. Figma is generated FROM code.

- **`.figma/manifest.json`** — Single source of truth for design identity: which Figma
  files to sync against, the font (Outfit), the icon set (Lucide, by component key),
  token namespace roots, text style naming patterns, and the variant property conventions
  every component set follows. Every Figma skill reads identity from here.
- **`/figma-ds`** (skill) — Reads a component's TSX (variants), CSS module (token usage),
  and metadata, applies the design identity from the manifest, and generates the matching
  Figma component set — variants named by convention, every fill and radius bound to
  the corresponding Figma variable. 6-step generation flow per component.
- **`/figma-composition`** (skill) — Generates Figma composition examples.
- **`/docs-to-figma`** (skill) — Bridges documentation to Figma pages.
- Supporting: `push-tokens.mjs`, `scan-variables.mjs`, `sync-components.mjs`,
  `maps/tokens.json`, `maps/components.json`, `Figma Variables`, `Figma Components`,
  `Figma Pages`, `Figma Transitions`, `.figma/schema/*.json`

### 🔴 6. Agentic Loop — "The loop that closes"
This is what makes the system self-healing rather than just self-documenting.

- **`/review-pr`** (skill, v1.2.0, governance gate) — Reviews pull requests for design
  system consistency: commit conventions, component structure, token usage, exports,
  metadata conformance, docs, accessibility, and atomic hierarchy. Scope-aware.
  Verdicts are severity-ranked: Blocker → Error → Warning → Info.
- **`Audit Reports`** — Token violations by component, severity-ranked, with suggested
  fixes. Because the infrastructure (index + maps) already knows every component and
  every token, generating a full-system report is a cheap, repeatable operation.
  Governance becomes a continuous artifact, not a quarterly consulting project.
- **`Self-healing suggestions`** — The agent doesn't just flag violations — it proposes
  the exact fix with rationale: which token replaces which hardcoded value, which
  canonical prop name replaces a drifted one. Loop: `audit → report → propose →
  (human approves) → heal`.

## Key Design Principles

1. **Token-first, always** — No hardcoded hex, px, or font values in any component CSS.
   Everything resolves to a `var(–token)`. If it doesn't, `/token-auditor` catches it
   and the loop proposes the fix.

2. **Anatomy is a contract, not a convention** — Every component has exactly the same
   5-file structure because `/scaffold-component` generates it. What the scaffold creates,
   `/review-pr` enforces.

3. **LLM decides WHEN; scripts decide WHAT** — The agent triggers indexing and auditing.
   Python scripts do the deterministic computation. This is why the system has 0.04%
   variance across runs (vs 26.5% without infrastructure).

4. **Code is the source; Figma is the render target** — The code (TSX variants + CSS
   module tokens) is the single source of truth. Figma is generated FROM code, not the
   other way around. Every fill in Figma is bound to a Figma Variable by durable key.

5. **Facts and judgment are separate nodes** — `scan-props.js` extracts facts
   deterministically. `build-prop-map.js` applies canonical judgment (axes, anti-synonym
   glossary, drift flags). Separated by design.

6. **Governance is continuous, not periodic** — Because the index knows everything,
   audit reports are cheap to regenerate. `/review-pr` runs on every PR. The loop never
   opens.

## Agent Operating Protocol

When Claude Code starts a session in this project:

1. **Load** `.ai/index.toon` — get the full map (78 components, 348 tokens,
   657 relationships). Do NOT crawl the codebase.
2. **Keep in context** — the index stays loaded for the conversation.
3. **Fetch on demand** — load individual `Component.metadata.ts` only when working
   on that specific component.
4. **Act through skills** — never improvise component creation or token changes
   outside of the skill system.
5. **Let rules shape you** — `.claude/rules/` rules load automatically by path.
   Read them, follow them.
6. **Close the loop** — after scaffolding or modifying, run `/codebase-index` to
   update maps, and `/review-pr` before any PR.
