# Agentic Design System — 22 Nodes

> "This isn't a pipeline — it's several systems running side by side and wired together.
> Tokenization and intent are their own systems; indexing interconnects them;
> orchestration wraps them; Figma plugs in as a satellite. The same grammar recurs
> in each: agency (skills that orchestrate), determinism (scripts and schemas that
> compute and constrain), state (the artifacts that result). Agentic doesn't mean
> everything is an LLM — let's tour the systems."
>
> — Cristian Morales Achiardi

This document describes the 22 nodes of the Agentic Design System architecture
that Protonic is built on. Each node shows: what it is, what it connects to,
and its status in Protonic.

---

## Layer 1 — Tokenization

*Where everything starts. Design decisions encoded as platform-agnostic contracts.*

### 1 · `tokens/*.json` — W3C design tokens

The foundation. Design decisions encoded as W3C Design Tokens (DTCG format) with
`$value` and `$type` on every entry. Two-layer color architecture: foundation
palettes referenced by semantic roles.

```json
// semantic-colors.json (excerpt)
{
  "color": {
    "primary": { "$value": "{color.orange.500}", "$type": "color" },
    "danger":  { "$value": "{color.red.600}",    "$type": "color" }
  }
}
```

**Protonic:** `tokens/` — 12 categories: color/foundation, color/semantic,
spacing, typography, radius, motion, elevation, text-styles, borders, z-index,
icon-sizes, breakpoints.

**Connects to:** Style Dictionary (compiles), `/codebase-index` (indexes),
`tokens-system.md` rule (governs naming).

---

### 2 · Style Dictionary — token compiler

Reads the W3C source JSON and emits platform outputs. CSS custom properties for
web, JS constants with TypeScript declarations for programmatic use. Runs before
any package that depends on tokens — components can never build against stale values.

```
tokens/*.json  →  Style Dictionary  →  variables.css
                                    →  tokens.js + tokens.d.ts
```

**Protonic:** `sd.config.mjs` — `usesDtcg: true`, `outputReferences: true`
(semantic tokens reference foundation vars, not baked hex).

**Connects to:** `tokens/*.json` (source), `variables.css` (CSS output),
`src/tokens/tokens.js` (JS output).

---

### 3 · JS Token Constants — programmatic token output

The JS/TypeScript output from Style Dictionary. Used by audit scripts, Figma
sync scripts, and any tool that needs to read token values programmatically.

```ts
// src/tokens/tokens.js (auto-generated)
export const colorPrimary = '#a43700';
export const spacer4 = '4px';
```

**Protonic:** `src/tokens/tokens.js` — auto-generated by `sd.config.mjs`.

**Connects to:** Style Dictionary (generates), `design-review.ts` (nearest-token
resolver reads spacing scale), `maps/tokens.json` (Figma binding seed).

---

### 4 · `variables.css` — compiled CSS custom properties

The compiled stylesheet every component references. Change a token at the source
and all components follow. Token names flatten to CSS custom properties.

```css
:root {
  /* foundation */
  --color-orange-500: #a43700;
  /* semantic — references foundation */
  --color-primary: var(--color-orange-500);
  /* scale */
  --space-4: 16px;
  --radius-pill: 100px;
}
```

**Protonic:** `src/tokens/variables.css` — auto-generated, do not edit directly.

**Connects to:** Style Dictionary (generates), `Component.module.css` (consumers).

## Layer 2 — Intent

*Components declare their identity through contracts, not code.*

### 5 · `Component.module.css` — token-based CSS module

Every visual decision resolves to a design token. CSS modules keep styles scoped
per component while `var(--…)` references tether them to the tokenization layer.
This is the surface the token auditor patrols.

```css
.chip {
  gap:           var(--space-1);
  font-family:   var(--font-family-body);
  border-radius: var(--radius-pill);
  transition:    background-color var(--motion-duration-fast) var(--motion-easing-out);
}
```

**Rule:** No hardcoded hex, rgb, px, font-family, or border-radius values.
Foundation tokens (`--color-orange-500`) must not appear in component CSS — use
semantic aliases (`--color-primary`) only.

**Connects to:** `variables.css` (token source), `/token-auditor` / `design-review.ts`
(audits), `/scaffold-component` (generates).

---

### 6 · `Component.tsx` — CVA variants + typed props

Component logic declares its variant API with class-variance-authority. Every prop
axis (variant, size, status) maps to CSS module classes with defaults. The TypeScript
interface is the contract that metadata, docs, the prop glossary, and Figma generation
all read from. One source, four consumers.

```tsx
const chipVariants = cva(styles.chip, {
  variants: {
    status: { info: styles.info, success: styles.success,
              warning: styles.warning, error: styles.error },
  },
  defaultVariants: { status: 'info' },
});
```

**Connects to:** `axes.ts` (prop vocabulary), `.ai/prop-map.md` (governed vocab),
`/figma-ds` (reads variants), `/scaffold-component` (generates).

---

### 7 · `/scaffold-component` — generates complete component anatomy

Components are not written from memory — they're scaffolded. One invocation
produces the same 8-file anatomy every time. What the scaffold creates, the
governance gate enforces.

```
src/components/Button/
  Button.tsx           # CVA + React Aria primitive
  Button.types.ts      # TypeScript interface
  Button.styles.ts     # CVA variant config
  Button.module.css    # token-based CSS
  Button.metadata.ts   # AI-ready metadata
  Button.stories.tsx   # Storybook coverage
  Button.test.tsx      # Vitest browser tests
  index.ts             # barrel exports
```

**Connects to:** `axes.ts` (canonical props), `metadata-schema.md` (schema),
`atomic-hierarchy.md` (category rules), `/review-pr` (audits output).

---

### 8 · `Component.metadata.ts` — AI-ready metadata

Structured, machine-readable documentation per component: use cases, composition
patterns, anti-patterns with alternatives, accessibility notes, AI hints, token
list, and Figma binding. This is the "API docs for agents" — it tells an AI when
and how to use a component correctly, and what NOT to do.

```ts
export const ChipMetadata = {
  id: 'chip', name: 'Chip', version: '1.0.0',
  type: 'interactive', category: 'atom',
  path: 'src/components/Chip',
  useCases: [...], antiPatterns: [...],
  a11y: { inherited: [...], owned: [...] },
  aiHints: [...],
  tokens: { background: [...], radius: [...] },
  figma: { fileKey: '...', component: 'Chip', properties: {...} },
};
```

**Connects to:** `/scaffold-component` (generates), `/docs-agent` (reads for
Storybook), `.ai/index.json` (indexed), `/review-pr` (checks conformance).

---

### 9 · Metadata Schema — versioned contract for metadata shape

The standardized schema every `Component.metadata.ts` must follow. Schemas turn
generated output into validatable output — a file that drifts is mechanically
detectable, not a matter of opinion.

**Protonic:** `.claude/rules/components/metadata-schema.md`

Required fields: `id`, `name`, `version`, `type`, `category`, `description`,
`path`, `created`, `modified`.
Valid `type` values: `interactive | display | container | input`.
Valid `category` values: `atom | molecule | organism | layout`.

**Connects to:** `/scaffold-component` (generates against), `/review-pr` (checks
conformance).

---

### 10 · Prop Glossary — canonical prop vocabulary + drift flags

The governed prop vocabulary across all components: an axes registry with canonical
values, an anti-synonym value glossary, per-prop usage, and drift flags.
Facts and judgment are separate nodes: a scan script extracts facts deterministically;
the map file applies canonical judgment. The Figma manifest's variant conventions
mirror these axes — one contract enforced on both surfaces.

**Protonic:** `.ai/prop-map.md`

```markdown
| Axis    | Concept            | Canonical values                  |
|---------|--------------------|-----------------------------------|
| variant | Component identity | primary / secondary / destructive |
| size    | Ordered scale      | small / medium / large            |
| status  | System feedback    | success / warning / error / info  |
```

**Connects to:** `axes.ts` (source of truth), `/scaffold-component` (reads vocab),
`.figma/manifest.json` (mirrors conventions), `/review-pr` Layer 2 (drift check).

## Layer 3 — Indexing

*The indexing layer maps relationships so the agent navigates instead of explores.*

### 11 · `/codebase-index` — generates relationship and dependency maps

Automatically generates relationship and dependency maps for the entire component
system. Runs the deterministic script, then loads the fresh index into agent context.

```bash
/codebase-index
# Outputs:
#   .ai/index.json                              project summary
#   .ai/relationships/component-usage.json      component graph
#   .ai/relationships/design-tokens.json        token usage map
#   .ai/relationships/dependencies.json         dependency chains
```

**Connects to:** `.ai/index-components.mjs` (runs), `.ai/index.json` (writes),
`.ai/relationships/*.json` (writes).

---

### 12 · `index-components.mjs` — deterministic indexer

The skill orchestrates; this script computes. A pure scanner that walks every
component, extracts React Aria foundation, CVA variants, CSS token references,
cross-component `usedBy` relationships, category, Figma binding, and contract
refs — then emits JSON maps. Same input, same output, every run. The LLM decides
WHEN to index; this script decides WHAT the index contains.

```bash
node .ai/index-components.mjs
# No LLM in the loop.
# Run it twice → identical output.
# (0.04% ARC variance comes from this split.)
```

**Connects to:** `/codebase-index` (triggered by), `.ai/index.json` (writes),
`.ai/relationships/*.json` (writes).

---

### 13 · `.ai/index.json` — the system index

The project summary the agent loads once at session start. Contains a `summary`
block (component count, token count, category count, relationship count), a
`usage` block (agent operating protocol), and the full component array.

```json
{
  "summary": {
    "totalComponents": 10,
    "totalTokens": 120,
    "tokenCategories": 12,
    "relationshipsMapped": 42
  },
  "usage": {
    "loadOnce": "Read index and relationships/* once at session start",
    "loadMetadataOnDemand": "Read *.metadata.ts only for the component you're working on"
  },
  "components": [...]
}
```

**Connects to:** `index-components.mjs` (generates), AI Agent (loads at start).

## Layer 4 — Orchestration

*Instructions tell the agent how to operate; rules shape how it works; skills give it verbs.*

### 14 · AI Agent — the operator

Claude Code is the runtime of the system. At session start it loads CLAUDE.md
(instructions) and `.ai/index.json` (the map), then operates exclusively through
skills. Rules constrain it. The index informs it. Every skill reads and writes
well-defined artifacts. The agent never improvises against the system.

**Agent Operating Protocol:**
1. Load `.ai/index.json` + `.ai/relationships/*.json` at session start
2. Keep the index in context for the whole conversation
3. Load `Component.metadata.ts` on demand only
4. Act through skills — never improvise component creation or freehand CSS
5. Let `.claude/rules/` shape behavior when touching relevant files
6. Close the loop: scaffold → audit → design-review → index

**Protonic:** `CLAUDE.md`

---

### 15 · Rules — macro-level context, auto-loaded by path

The registry of project-specific rules in `.claude/rules/`. Each rule carries a
`paths:` frontmatter so it loads only when the agent touches matching files.
Rules don't enforce like code — they teach the agent the what/how/why of each
system it operates in.

```
.claude/rules/
  tokens-system.md          # loads on: tokens/**, src/tokens/**
  components/
    metadata-schema.md      # loads on: src/components/**/*.metadata.ts
    atomic-hierarchy.md     # loads on: src/components/**
  codebase-index/
    rules.md                # loads on: .ai/**
  skills-integration.md     # loads on: .claude/**
  best-practices.md         # loads on: src/**
```

**Connects to:** AI Agent (shapes behavior), all systems (governs each).

## Layer 5 — Figma Integration

*Code is the source. Figma is the render target.*

### 16 · `.figma/manifest.json` — design identity source of truth

Single source of truth for design identity: which Figma files to sync against,
fonts, icon set (with durable component key), token namespace roots, text style
naming patterns, and the variant property conventions every component set follows.
Every Figma skill reads identity from here.

```json
{
  "fileKey": "R4KIv7XKg8g1A1NR9R11cI",
  "fonts": { "display": "Bricolage Grotesque", "body": "Plus Jakarta Sans" },
  "iconSet": { "name": "Lucide", "componentKey": "TBD" },
  "textStyleNaming": { "pattern": "{category}.{size}.{weight}" },
  "tokenNamespaces": { "roots": ["color-primary", "color-text", "color-bg"] },
  "variantConventions": { "properties": { "Variant": [...], "Size": [...] } }
}
```

**Connects to:** `.ai/prop-map.md` (mirrors vocab), `/figma-ds` (reads identity),
`figma-cli daemon` (uses file key).

---

### 17 · `/figma-ds` — generates Figma component sets from source

Reads a component's TSX (variant axes), CSS module (token references), and metadata,
applies the design identity from the manifest, and generates the matching Figma
component set — variants named by convention, every fill bound to the corresponding
Figma variable by durable key.

```
/figma-ds Button

# 1. Read Button.tsx        → variant axes (primary/secondary/destructive, sm/md/lg)
# 2. Read Button.module.css → var(--color-primary), var(--radius-pill)
# 3. Read manifest.json     → font Bricolage Grotesque, naming conventions
# 4. Resolve maps/tokens.json → bind fills to Figma variables by key
# 5. Generate component set via figma-cli daemon
# 6. Write back componentSetId + componentKey to maps/components.json
```

**Connects to:** `Component.tsx` (reads variants), `.figma/manifest.json` (identity),
`.figma/maps/tokens.json` (variable binding), `.figma/maps/components.json`
(write-back), `figma-cli daemon` (generates).

## Layer 6 — Agentic Loop

*The loop that closes. The system is self-auditing and self-healing.*

### 18 · `/review-pr` — governance gate on every change

Reviews changes for design system consistency across 7 layers: file structure,
axes registry compliance, React Aria conventions, token compliance, metadata
conformance, export completeness, and Storybook coverage. Checks scope to what
actually changed. Verdicts are severity-ranked with skipped checks reported.

```
/review-pr src/components/Chip

[ERROR]   Chip.module.css uses foundation token var(--color-red-600)
          → use semantic alias var(--color-danger)
[WARNING] Chip.metadata.ts missing antiPatterns array
[SKIPPED] Storybook checks — no .stories.tsx in diff
```

Severity: **Blocker → Error → Warning → Info**

**Connects to:** `metadata-schema.md` (checks conformance), `axes.ts` (drift check),
`design-review.ts` (token check), audit reports (verdicts).

---

### 19 · Audit Reports — token violations, severity-ranked

The audit output: token violations by component, severity-ranked, with suggested
fixes. Because the infrastructure (index + maps) already knows every component
and every token, generating a full-system report is a cheap, repeatable operation.
Governance stops being a quarterly project.

```markdown
# Token Audit — Protonic

| Component | Violations | Severity |
|-----------|-----------|----------|
| Chip      | 0         | —        |
| Button    | 0         | —        |

All components clean. Last run: 2026-06-24
```

**Protonic:** `src/agents/audit-all.ts` generates `audit-report.md` at project root.

**Connects to:** `index-components.mjs` (knows all components), `design-review.ts`
(runs per component), `/review-pr` (verdicts feed in).

---

### 20 · Self-healing suggestions — exact fixes with rationale

Because design decisions are encoded as explicit tokens and contracts, the system
can do more than flag violations — it proposes the exact fix: which token replaces
which hardcoded value, which canonical prop name replaces a drifted one. The agent
closes the loop: audit → report → propose → (human approves) → heal.

```diff
/* Chip.module.css */
-.dismiss { transition: opacity 200ms ease; }
+.dismiss { transition: opacity var(--motion-duration-normal) var(--motion-easing-out); }

// Rationale: 200ms === motion.duration.normal
//            ease is not a token — use var(--motion-easing-out)
```

**Connects to:** Audit Reports (source), AI Agent (proposes and enacts fixes).

### 22 · The loop, closed — overview

Step back and there's no top or bottom, no start or end — just systems running
together and closing back on themselves. The agent reads the indexes to find
everything and acts only through its skills; what those skills generate — maps,
audit reports, fix suggestions — loops back to the agent that consumed them.
That self-closing loop is what "agentic" means.

```
tokens/*.json
    ↓ Style Dictionary compiles
variables.css
    ↓ components reference
Component.module.css + Component.tsx
    ↓ /scaffold-component generates, /review-pr enforces
Component.metadata.ts
    ↓ /codebase-index maps
.ai/index.json + .ai/relationships/*.json
    ↓ AI Agent loads, navigates, acts through skills
/figma-ds → Figma (render target)
/review-pr → Audit Reports → Self-healing suggestions
    ↓ (human approves) → healed components
    ↑ loop continues
```
